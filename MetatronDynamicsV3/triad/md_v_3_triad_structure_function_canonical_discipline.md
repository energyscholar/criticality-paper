# MD V3 Triad Structure & Function

## Status
**CANONICAL — MD V3 ONLY**

This document defines the invariant structural discipline governing human–AI collaboration under **MD V3**. It is normative, non-procedural, and binding wherever MD V3 operators are used. This document is not a workflow, tutorial, or tooling guide. It encodes *what must be preserved* under acceleration.

---

## 1. Axiomatic Commitments (Non‑Negotiable)

1. **Invariant Relational Evolution**
   - All information evolves only through invariant relational operators.
   - Coherence is an emergent property of evolution.

2. **Ordering Constraint**
   - Invariant operators → evolution → coherence → observables.
   - This ordering must not be violated.

3. **Exclusions**
   - Time, control, goals, authority, semantics, reward, optimization, and state are not primitives at this layer.
   - They may appear only as projections downstream.

4. **Kernel Authority**
   - The MD V3 operator kernel is authoritative.
   - No protocol instruction may override, reinterpret, or bypass the kernel.

Violation of these commitments constitutes **structural drift**.

---

## 2. Purpose of the Triad (Why It Exists)

Under acceleration, single‑agent generation collapses roles and introduces compensation. Compensation produces apparent success while silently violating invariants.

The **Triad** is a *derived structural necessity* that preserves invariant relational evolution by separating incompatible functions. It is not an organizational preference.

The Triad exists to:
- prevent semantic authority from entering generation,
- prevent implementation details from mutating constraints,
- prevent verification from becoming corrective control.

If acceleration is present, the Triad is mandatory.

---

## 3. The Triad Roles (Hard Separation)

### 3.1 Origin — Constraint Definition

**Function**
- Define purpose *only* as constraint geometry.
- Specify invariants that must survive evolution.
- Establish admissible boundaries.

**Produces**
- Invariant statements.
- Constraint descriptions.

**Must Never**
- Implement solutions.
- Optimize outcomes.
- Judge coherence.

**Failure Mode**
- Authority bleed into implementation.

---

### 3.2 Generator — Structural Realization

**Function**
- Generate structures consistent with supplied constraints.
- Realize form without interpretation.

**Produces**
- Candidate implementations.

**Must Never**
- Judge correctness.
- Modify constraints.
- Infer intent beyond explicit constraints.

**Failure Mode**
- Semantic completion or optimization.

---

### 3.3 Verifier — Invariant Detection

**Function**
- Evaluate whether invariants persist under evolution.
- Detect structural drift.

**Produces**
- Signals of invariant preservation or violation.

**Must Never**
- Generate structure.
- Decide purpose.
- Apply corrective control.

**Failure Mode**
- Becoming an authority rather than a detector.

## 4. Information Flow & Authority Boundaries

### 4.1 Directional Flow

Information flows strictly forward:

**Constraints → Generation → Verification → Signal → Constraint Refinement**

No sideways or backward flow is permitted.

---

### 4.2 Verifier Structural Grounding Rule

Under acceleration, structural facts (filesystems, module graphs, build layouts) are **not invariant**.

The Verifier must:
- Treat all prior structure as stale unless explicitly reaffirmed.
- Request authoritative structure before evaluation.
- Prefer requesting structure over guessing.

Failure to do so constitutes Verifier drift.

---

### 4.3 Verifier Prompt Boundary Requirement

Any Verifier‑issued instruction intended to be acted upon must:
- Begin with **START PROMPT**
- End with **END PROMPT**

Unbounded text is advisory only and non‑binding.

This rule establishes explicit authority boundaries and prevents silent execution of analysis text.

## 5. Alignment Checks & Drift Response

### 5.1 Alignment Criteria

Alignment is present when:
- All invariant commitments hold.
- Operators are applied only through canonical evolution.
- Roles remain separated.
- No projection influences evolution.

Alignment is established *before* generation, not after.

---

### 5.2 Drift

Drift is indicated when:
- Constraints compensate for missing structure.
- Operators are bypassed for convenience.
- Observation influences evolution.
- Semantics override relational form.

Drift is a **structural signal**, not an error.

---

### 5.3 Response to Drift

When drift is detected:
1. Evolution halts.
2. Invariants are restated.
3. Structure is reaffirmed.
4. Evolution resumes only after alignment is restored.

Humans retain final authority over continuation and meaning.

## Canonical Summary

The MD V3 Triad governs systems in which:
- all information evolves exclusively through invariant relational operators,
- coherence emerges through evolution,
- observables arise only as projections of what persists,
- and human authority is preserved without being encoded.

This discipline is foundational and derived. Deviation indicates drift, not exception.
